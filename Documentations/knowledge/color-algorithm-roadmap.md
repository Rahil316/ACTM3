Material no longer inherits it.

**Update (2026-07-14): all hand-rolled color-space math has been replaced with library-backed implementations.** After two independent hand-rolled-CAM16 bugs (below) and a third pre-existing HCT forward-transform bug found while validating the replacement, and a fourth pre-existing OKLCH gamut-boundary instability found during the cutover's before/after token diff (all four listed in the confirmed-issues section below), the entire custom HCT/CAM16 implementation and the OKLCH conversion/gamut-chroma primitives in the engine were deleted and replaced by `src/shared/colorMath/` — OKLCH backed by `culori`, HCT backed by a vendored (not npm-dependency) copy of Google's own `material-color-utilities` `Hct` class, trimmed to the ~1,500 lines that class actually needs (`src/shared/colorMath/hct-vendor/`). `TONAL_SCALE_ALGO.Material`'s manual tone-search bisection loop was also deleted entirely — the vendored `Hct` class does correct, hue-safe tone-solving internally the moment `.tone` is set, so no search is needed at all; a direct closed-form CIE L*↔Y conversion (`lstarFromY`) plus a small ±2-tone local refinement replaces it. Verified end-to-end via a before/after diff of every real preset's full compiled token output (not just the raw conversion primitives) — see the confirmed-issues section for what that diff surfaced. `_relLumFromLinear`/`_wcagContrast`/`_lumOfHex` (a second, private luminance implementation used only inside `solveColorForContrast`'s hot bisection loop, now in `src/shared/engine/solverEngine.ts`) were deliberately **not** migrated — benchmarked directly, `culori`'s general-purpose `wcagLuminance` is ~7x slower per call than this tight inline version, and the loop calls it up to 60 times per color/role/variation/theme combination.

## 1. Current algorithm analysis

Two independent mechanisms exist, now split across two files (see the 2026-07-15 update above) — easy to conflate:

- **Scale algorithms** (`ScaleAlgorithm`: `Natural`, `Uniform`, `Linear`, `Expressive`, `Symmetric`, `OKLCH`, `Material`, `Fidelity`) — Scale mode, `src/shared/engine/clrEngine.ts`. Generate one full N-step tonal ramp per color; roles then pick steps off that ramp by contrast target (there is no index-based mapping method — see `Documentations/knowledge/how-it-works.md`).
- **Solver modes** (`SolverMode`: `natural`, `constant-chroma`, `symmetric`, `max-chroma`, `gamut-cusp`, `apca-natural`) — Direct mode, `src/shared/engine/solverEngine.ts`. Solve a single color directly for a target contrast ratio against a background, no precomputed ramp. (A seventh mode, `hue-locked`, was removed entirely 2026-07-15 — see the confirmed-issues entry below for its history.)

They solve a related problem ("how vivid should a color get as it moves away from its natural tone") with two unrelated, independently-tuned formulas — no shared "personality" between the two modes today.

### Scale algorithms, one by one

| Algorithm | Space | Mechanism | Verdict |
|---|---|---|---|
| **Linear** | HSL | Even HSL-lightness steps, fixed hue/sat | Worst of the seven — HSL lightness isn't perceptually uniform (mid-L yellow looks far brighter than mid-L blue). |
| **Uniform** | HSL | Binary-searches HSL-L for uniform real luminance steps, hue/sat held flat | Fixes spacing; flat saturation makes near-white/near-black steps look muddy or oversaturated. |
| **Natural** | HSL | Uniform + empirical saturation taper (`sat × (1 - (|L-50|/50)^1.5 × 0.4)`) | Reasonable patch, but the exponent/cap are hand-picked, not derived from any perceptual model. Current default. |
| **Expressive** | HSL | Natural + hue drift toward warm/cool at light/dark ends | Nice instinct, but hardcoded target hues (60°/240°) and factor (0.15) regardless of the seed's own hue family. |
| **Symmetric** | HSL (log-lum) | Anchors the seed's actual luminance at the ramp's middle index, interpolates independently on each side | Good, underused idea (most algorithms assume the seed sits at ~50% lightness) — but no saturation taper at all, and spacing gets uneven if the seed is itself very light or very dark. |
| **OKLCH** | OKLCH | Binary-searches L, holds chroma+hue exactly constant | Real perceptual space, genuine improvement — but **no gamut-mapping**. Channels silently clip near the ramp extremes for saturated seeds. |
| **Material** | HCT (Google Material You space) | Binary-searches tone, holds hue+chroma constant, gamut-maps via a full port of Google's `findResultByJ`/`bisectToLimit` solver | Most rigorous of the seven — real perceptual tone model + correct gamut handling, now genuinely hue-safe by construction (see confirmed-issues entry below — this was not always true). This is Google's own MD3 tonal palette algorithm. |

**Material is the best-engineered algorithm here, but the flawed HSL-based "Natural" is the shipped default.**

### Solver modes (Direct mode)

The contrast solver (`solveColorForContrast`) is the most solid part of the file: OKLCH-based, gamut-safe (`_maxChromaAtLH` binary-searches to the true in-gamut boundary), sane fallback (falls back to black/white with a clear warning when a target is mathematically unreachable), and reports precise diagnostics (`isAdjusted`, `chromaReduced`, `clipped`).

### Confirmed issues

- **`hue-locked` solver mode was functionally identical to `natural` — resolved by removal (2026-07-15).** Was `src/shared/engine/solverEngine.ts:233-243` (moved from the pre-split `clrEngine.ts` at the same relative logic — line number updated after the 2026-07-15 engine split, see status note above). Three independent sources disagreed about what it did — the Settings UI copy said "pushes to the maximum in-gamut chroma at each lightness"; a shipped dev preset comment said "picks light or dark text per color automatically"; the actual code computed `rawC` via the exact same `_targetChroma(..., "natural")` curve as `natural` mode and clamped it the same way. The resulting color was identical to `natural`'s — only an internal diagnostic-flag threshold differed — numerically confirmed byte-identical aggregate quality metrics across a full stress-test run before removal (0.520 vividness-preserved, 6.61% went-gray, to three significant figures, both modes, 29,600 tokens each). Read like an abandoned/half-finished mode, so rather than implementing the promised-but-never-built "push to gamut maximum" behavior or repointing the misleading copy, it was deleted from `SolverMode` outright — no successor value, no migration needed (any preset that had set `solverMode: "hue-locked"` would need updating, though none did at time of removal). **`gamut-cusp`** (`solverEngine.ts:118-120`, `_gamutRelativeChroma`) remains this codebase's real "push to gamut maximum, gamut-relative" solver mode, and is the closest existing precedent if `hue-locked`'s originally-stated intent is ever revisited as a new mode.
- **Per-Role scale-algorithm scope is dead in Scale mode.** `_generateScales` (`src/shared/engine/clrEngine.ts:313-343`) reads only `color.scaleAlgorithm`, never `role.scaleAlgorithm` — anywhere in the pipeline. But the field is fully wired end-to-end: `RoleGroupCard.tsx:166` renders a live per-role "Algorithm" dropdown (shown when Scale mode + "Uniform Algorithm" off + scope = "Per Role"), it's persisted, exported through `figma/config.ts:136`, and shown in the dev inspector's ConfigTree. Users can set it, believing it does something — it's silently ignored. Structurally this makes sense (a scale is generated once per color; two roles referencing the same color's scale can't each get their own ramp under the current one-scale-per-color data model), but the UI doesn't communicate that limitation. Direct mode's equivalent scope control works correctly (`_getSolverMode`, `clrEngine.ts:345`, does read `role.solverMode`) — the bug is specific to Scale mode.
- **OKLCH scale algorithm has no gamut-mapping** — unlike Material, which bisects to the true in-gamut boundary via `hctToHex`.
- **Zero test coverage on `clrEngine.ts`.** The `tests/` directory was removed entirely in a later cleanup pass (deprecated vitest/storybook infrastructure) and has not been reinstated — there is currently no test file anywhere in the repo, let alone one covering `scaleMaker`, `variableMaker`, `solveColorForContrast`, `hexToOklch`/`oklchToHex`, or `hexToHct`/`hctToHex`. The ~900 lines doing the actual perceptual math remain completely unguarded. Directly implicated in how long the HCT bug below (and the `Fidelity` gamut-envelope quirk above it) went unnoticed, and in how much manual numerical verification was needed to confirm the fix below rather than a quick test run.
- **`Material`'s HCT gamut solver produced wrong-hue swatches for saturated seeds — confirmed, root-caused, and fixed (2026-07-14).** Discovered while building the `nMobile` preset: Gray (`#71717A`) and a spare cyan (`#06B6D4`) both showed one or more darkest-step swatches with a hue completely unrelated to the seed (Gray's step 20 read `#2e0049`, a saturated purple, between neutral-gray neighbors; the cyan's steps 19–21 read as vivid violet after a correct dark-blue step 18). Root cause, confirmed against Google's own `material-color-utilities` reference implementation (`hct_solver.ts`, `cam16.ts`): two independent defects in `_hctRgbOrNull`/`_x2hct`. (1) The CAM16 hue-eccentricity factor `e_t` (`eHue = 0.25 * (cos(hueRadians + 2) + 3.8)`) was omitted entirely from both the forward (`_x2hct`) and inverse (`_hctRgbOrNull`) transforms' `p1` term — this codebase hardcoded `p1 = (50000/13) * Nc * Ncb` as if hue-independent, when it should be `eHue * (50000/13) * Nc * Ncb`. This mislocated a genuine pole in the CAM16 inverse's chromatic denominator for some hues. (2) Even after correcting `eHue`, the pole's existence is inherent to this parameterization for high-chroma seeds at low tone — `hctToHex`'s search re-bisected requested chroma and re-ran the same (still occasionally unstable) CAM16 inverse at every step with no hue-validity check on the result, so it could silently accept a mathematically-valid-but-wrong-hue in-gamut answer. The reference implementation never has this problem because it doesn't work this way: `findResultByJ` does 5 rounds of Newton's method directly on `J` (no chroma-search loop to wander off in), and only when that fails to converge does it fall back to `bisectToLimit` — a geometric RGB-gamut-boundary search that measures actual hue at every candidate (`hueOf`) and only accepts points confirmed to sit between the known-good and target hue, so it cannot produce a wrong-hue result by construction. **Fix**: added the missing `eHue` factor to both transforms, then replaced `hctToHex`'s chroma-bisection body with a full, faithful port of `findResultByJ` + `bisectToLimit` and their full dependency chain (`_CRITICAL_PLANES` 255-entry gamut-plane table, `_hueOf`, `_areInCyclicOrder`, `_bisectToSegment`, `_nthVertex`, etc.), adapted from the reference's 0–100 scale to this file's existing 0–1 linear-RGB convention. Verified via direct numerical stress test: ~15,000 `(hue, chroma, tone)` samples across 24 hues × 7 chroma levels × ~91 tone steps show no more wrong-hue-family jumps (worst residual dropped from 70–100°+ flips to a max ~20° seam at the primary/fallback-solver handoff boundary, consistent with the pre-existing "20-45° hue drift" already noted above as an inherent, accepted characteristic of this HCT implementation — not a regression). Both original reproductions (Gray, the cyan spare) now produce fully smooth, monotonic, hue-consistent 21-step ramps.
- **Naming collision**: "Natural" is both a `ScaleAlgorithm` value and, lowercased, a `SolverMode` value. Different mechanisms, same word, shown in adjacent Settings panels depending on plugin mode.
- **Four of the seven HSL-family scale algorithms (`Natural`, `Uniform`, `Expressive`, `Symmetric`) target a hue-dependent, not hue-uniform, luminance step series — confirmed and quantified (2026-07-14).** This is the same root defect the "Linear" row above gestures at informally ("mid-L yellow looks far brighter than mid-L blue"), but it also silently affects the four algorithms marketed as perceptually uniform, not just the one (`Linear`) that admits it isn't. All four call `scaleMaker`'s shared `stepLum(i)` (`engine/clrEngine.ts:262-265`) to get a *target relative luminance* for step `i` — identical across every seed color regardless of hue — then binary-search via `findL` (`engine/clrEngine.ts:267-280`), which measures progress with `relLum(hslToHex(...))`, i.e. the plain WCAG sRGB relative-luminance formula (`0.2126·R + 0.7152·G + 0.0722·B`). That formula is not hue-uniform: at fixed HSL lightness/saturation, green- and yellow-hued colors have dramatically higher relative luminance than blue-hued ones, because the coefficients weight green ~10x and red ~3x more heavily than blue. Since the *target* is luminance (hue-blind) but the thing actually being reported as "the step" is HSL lightness (hue-sensitive), different-hued seeds land at very different perceptual positions for the same nominal step index. **Reproduced numerically**: replicating `Natural`'s exact `stepLum`/`tapS`/`findL` formulas for a 9-step scale, a blue seed (hue 220°) and a yellow seed (hue 50°), both sat 70, diverge fast — HSL-L at step 2 is 73.3 (blue) vs. 46.0 (yellow), a 27-point gap; by step 4 blue is still at 54.8 while yellow has already collapsed to 32.2. The yellow ramp is visibly crammed dark by step 4-5 (`#86751d`, `#6b5d1a` — muddy olive) while the blue ramp still has clear light-to-mid steps remaining at the same indices. This directly reproduces Ant Design's documented reason (`@ant-design/colors`) for using an asymmetric, hue-dependent hue-rotation/saturation curve rather than one uniform formula for all seeds — this codebase has no such compensation for the four affected algorithms. **Not affected**: `OKLCH`, `Material`, and `Fidelity` also call `stepLum(i)` for their target, but binary-search in OKLCH `L` or HCT tone respectively (both genuinely hue-normalized perceptual coordinates) rather than HSL lightness measured via raw sRGB luminance, so they don't inherit this skew. `Linear` doesn't call `stepLum`/`findL` at all (even HSL-L steps, no luminance target), so it's a different, already-disclosed failure mode, not this one. **Practical implication**: `Natural` (the shipped default) and its three siblings will produce systematically worse, dark-crowded ramps for yellow/lime/warm-green seeds specifically, and better-spread ramps for blue/violet seeds, with no warning surfaced to the user. **Not yet fixed** — documented in `Documentations/knowledge/color-system-guidelines.md`'s Algorithm Selection Guide as a user-facing caveat (recommends `OKLCH`/`Material`/`Fidelity` over the four affected algorithms for warm seeds) pending an engine-level fix. Any real fix converges with the "gamut-relative chroma" idea in §2 below — replacing HSL+raw-luminance with an OKLCH/HCT-native `stepLum`/`findL` pair for all four algorithms, not a per-algorithm patch — since the underlying defect is the shared `scaleMaker` luminance-targeting scaffold, not any one algorithm's individual formula.
- **`Fidelity`'s lightest 1–2 steps can look abruptly desaturated for some hues — confirmed real, not a bug, and root-caused (2026-07-13).** Reported while building the `nMobile` preset (`src/shared/presets/raw/dev/nmobile.ts`) — the Teal spare swatch showed a visibly sudden jump toward white in its last couple of steps. Traced precisely: `stepLum`'s log-luminance series (`engine/clrEngine.ts:262-265`) allocates its lightest step(s) very close to `Y=1` (e.g. a 21-step scale's lightest target is ~95% of max luminance). At that extreme, every hue's true in-gamut chroma envelope (`_maxChromaAtLH`) has almost collapsed to zero — this was confirmed numerically across five seed hues (teal, green, purple, indigo, yellow): each hue's max-chroma-vs-lightness curve peaks somewhere around `L≈0.85–0.92` (varies by hue) then falls off steeply toward `L=1`, where chroma is genuinely near-zero for *any* hue (that's real colorimetry — white has no chroma by definition). `Fidelity` is doing exactly what its design promises (following the true envelope, `f × envelope(L)`), so this isn't a defect in the algorithm's logic. The reason it looks "sudden" rather than gradual is that the step spacing crowds 1–2 steps into the narrow lightness band where the envelope is steepest; other algorithms (`Natural`, etc.) never show this because they apply a synthetic smooth saturation taper instead of following the real gamut boundary, so they're consistently smooth but not gamut-accurate at the light end. **Severity is hue-dependent**: hues whose envelope peak sits below/near their own seed lightness (purple, indigo, red — peak around `L≈0.5–0.63`) taper gradually the whole way and never show a visible cliff; hues whose peak sits well above the seed (teal, green — peak around `L≈0.87–0.90`) have more of their light end past the peak, where the fall is steepest, producing the visible jump.
  - **Investigated fix, rejected as too costly**: floor-clamping the last step's chroma to a fraction of the next step's chroma (re-solving for the nearest lightness where that floor is achievable) does smooth the visual transition — verified numerically, e.g. teal's lightest step went from chroma 0.028 to 0.075, clearly less washed out — but costs a real luminance deviation (~7% relative luminance on that one step) from its intended contrast-target position, undermining the log-luminance series' whole point of even perceptual spacing for that step. Decided not worth the risk/complexity for a single-step cosmetic fix; left as documented, expected behavior instead of an engine change. **Superseded (2026-07-14)**: this cliff was a real gamut-physics limit (correctly identified above), but the hand-rolled `_maxChromaAtLH` measuring that gamut boundary turned out to have its own separate instability near white (see the next entry) — the library-backed replacement's boundary is smooth, so the cliff itself is now gentler for most hues even though the underlying physical limit (near-zero chroma at near-white lightness) still applies and can't be engineered away.
- **A second, independent hand-rolled-CAM16-style instability existed in the OKLCH gamut-boundary search (`_maxChromaAtLH`/`_inGamutOklch`) — found during the library-migration cutover's before/after token diff, fixed by the migration itself (2026-07-14).** Not found by this session's earlier direct numerical investigation of `Fidelity`'s light-end cliff (above) — that investigation confirmed the *cliff's existence* was real physics, but never stress-tested whether the *specific bisection search* measuring the gamut boundary was itself numerically stable near white. It wasn't: direct comparison of the old hand-rolled `_maxChromaAtLH` against a fine `L` sweep near `L≈0.85–0.999` found 2 of 36 sampled hues (yellow-green ≈90°, blue-cyan ≈210°) where reported max chroma *increased* after `L` passed the true envelope peak — a physically impossible result (chroma capacity cannot recover as lightness approaches white) — almost certainly an 8-bit-hex-quantization round-trip artifact in the old `_inGamutOklch`'s `hexToOklch(oklchToHex(...)).C >= C - 0.002` gamut check, similar in character to the CAM16 near-black quantization instability found earlier in the `Material` investigation, just manifesting near white in OKLCH instead of near black in HCT. Confirmed via the library cutover's real-preset before/after diff: `nMobile`'s `Fidelity`-generated Attention swatch and `nclarity`'s Direct-mode `Status/Attention` background token (both yellow-hued, both solving for a near-white/low-contrast target) showed genuinely large (100+ of 255) single-channel jumps, concentrated exclusively at low contrast targets (1.03–3) — exactly where the search lands closest to the unstable near-white region. The `culori`-backed replacement (`src/shared/colorMath/oklch.ts`'s `maxChromaAtLH`) showed zero non-monotonic cases across a finer 72-hue sweep of the same region. Not a regression from the migration — a second latent bug the migration surfaced and fixed as a side effect, on top of the intentional `Material` fix.
  - **If revisited**: any real fix belongs with the "gamut-relative chroma" work in section 2 above (e.g. a "Taper Reference"/envelope-aware knob in the section 3 equalizer idea), not a one-off patch to `Fidelity` alone, since the underlying tension (exact contrast-step luminance vs. real gamut chroma limits near white) is shared by any algorithm that tries to be simultaneously gamut-accurate and log-luminance-uniform.

### Verdict

- **Accessibility**: strong. Contrast-target-driven mapping is the backbone of both modes; the solver has real gamut-awareness and honest failure reporting.
- **Aesthetics**: uneven. One algorithm (Material) is done right, one (OKLCH) is nearly right with a fixable gap, the other five are a legacy HSL family papering over HSL's unsuitability for tonal ramps with increasingly specific empirical patches rather than a shared model.

## 2. The "gamut-relative chroma" idea (the "ultimate" algorithm)

**Update (2026-07-15): this idea has now shipped twice, not once.** It shipped for Scale mode as `Fidelity` (as this section already describes below). It has *also* now shipped for Direct mode, as the `gamut-cusp` and `apca-natural` solver modes (`src/shared/engine/solverEngine.ts`'s `_gamutRelativeChroma`, line 102) — same `f = srcChroma / envelope(srcTone)`, held constant, `target = f × envelope(L)` mechanism described below, just applied inside the Direct-mode bisection instead of a fixed scale-step series. `apca-natural` additionally swaps the *search metric* from WCAG contrast to APCA Lc (see status note at the top of this doc) — the chroma-shaping idea itself is identical between the two new solver modes. The open question this doc originally posed — "run it in HCT or OKLCH" — was answered the same way in both places: OKLCH, reusing `maxChromaAtLH` (this doc's `_maxChromaAtLH`/`hctToHex` references below are now historical; see the library-migration update above).

Core insight nothing here exploits: **every hue has its own uniquely-shaped maximum-chroma envelope across the lightness range**, asymmetric per hue (yellow's max chroma peaks near the light end; blue's peaks mid-dark). No algorithm here — and no mainstream design-tool ramp generator known — builds a ramp aware of this envelope's *shape* per-hue. They either hold raw chroma constant and clip on contact (OKLCH here, effectively Material/Tailwind too) or apply a universal taper curve blind to the hue's real gamut (Natural/Expressive here).

**Mechanism, three parts:**

1. **Compute the seed's chroma as a fraction of its own hue's envelope, at its own lightness.** Sample max-in-gamut chroma at the seed's hue+tone (one bisection — reuses `_maxChromaAtLH`/`hctToHex`'s existing bisection primitive). `f = srcChroma / envelope(srcTone)` — "how close to maximally vivid is this color, for what's physically possible at its hue/tone."
2. **Hold `f` constant across every step, not raw chroma.** For each target lightness, sample that lightness's envelope for the same hue, target chroma = `f × envelope(L)`. The taper *is* the hue's real gamut shape, scaled by the seed's own vividness — not a guessed universal exponent.
3. **Gamut-safe by construction, no fallback needed.** Since `f ≤ 1` always (the seed is a real achievable color), `f × envelope(L') ≤ envelope(L')` at every step. No bisect-down-to-fit needed anywhere — Material's gamut-correctness and OKLCH's "stay true to the seed" goal simultaneously, which are mutually exclusive today.

**Run it in HCT, not OKLCH**, for the hue axis — HCT already corrects for the Hunt/Bezold–Brücke hue-perception shift OKLCH doesn't, and `hctToHex`'s bisection is most of what's needed for envelope sampling already (reuse, not new color math).

**Why it beats each precedent:**
- vs. **Material** (raw chroma held, clip-on-demand): fixes "some hues clip hard, others look understated" — every seed gets consistent *relative* vividness regardless of hue.
- vs. **OKLCH** (raw chroma, no gamut-map at all): same fix, plus closes the channel-clipping bug.
- vs. **Natural/Expressive** (arbitrary taper curves): taper is the real physical gamut boundary, not a guessed exponent.
- vs. **Symmetric** (anchors lightness, no chroma taper): complementary, not competing — combine Symmetric's "don't assume the seed sits at 50%" with this chroma idea into one algorithm.

**Open sub-question**: should the seed's exact hex be forced to appear as one literal ramp step (guaranteed fidelity, tiny local unevenness around that step) or only used to calibrate `f` (perfectly smooth curve, seed itself may not be pixel-identical to any step)? Material doesn't force literal inclusion either. **Resolved as a spectrum, not a binary** — see Seed Anchor knob below; different presets can sit at different points on it.

## 3. The algorithm equalizer idea

Instead of shipping this as one more preset in a dropdown, decompose all seven existing algorithms (+ the new idea) into the orthogonal knobs they're each secretly hard-coding one fixed combination of. No mainstream design tool or Figma plugin exposes a perceptual color algorithm's internal parameters as a live, tunable equalizer — this is a genuinely novel product surface, not just an algorithmic tweak.

| Knob | Controls | Where existing presets sit |
|---|---|---|
| **Seed Anchor** (0→1) | Force the seed's own lightness/hex to sit at its real relative position vs. ignore it for a pure uniform curve | Uniform/Natural/OKLCH/Material = 0. Symmetric = 1 (crudely). Top end = literal seed inclusion, resolving the open sub-question above. |
| **Taper Amount** (0→1) | How much chroma falls off toward light/dark extremes | Uniform/OKLCH = 0 (flat). Natural/Expressive = fixed ~0.4. Nobody offers "aggressive" or "none, with warning." |
| **Taper Reference** | Absolute chroma vs. gamut-relative fraction (idea #2) | Every existing algorithm = absolute. Gamut-relative is the new option. |
| **Hue Drift** (0→1 + direction) | Warm/cool hue shift at light/dark ends | Expressive = fixed ~0.15 toward hardcoded 60°/240°. Everyone else = 0. |
| **Perceptual Model** | HSL / OKLCH / HCT | Linear/Uniform/Natural/Expressive/Symmetric = HSL. OKLCH = OKLCH. Material = HCT. |
| **Contrast-Aware Placement** (on/off) | Nudge step lightness toward the actual configured variation targets instead of pure uniform spacing | Nobody does this today — the direct link to "accessible + pleasing." |

**Rules for this feature:**
- **Gamut correctness is never a knob.** Always mapped correctly; no user-facing "let it clip" option — there's no aesthetic upside to that, only accidental breakage.
- **Warnings are triggered by output sanity checks, not input-range thresholds.** Catches bad *combinations* a per-knob range limit can't predict:
  - Step collision — two steps landing perceptually identical.
  - Chroma collapse — most steps going near-gray unintentionally (some hues have a narrow high-chroma window in sRGB; gamut-relative taper can expose that harshly).
  - Hue drift beyond recognition — lightest/darkest steps reading as a visibly different hue family than the seed.
  - Broken monotonicity in lightness/contrast — this one is a correctness guard to **enforce**, not just warn about, since `_mapByScaleContrast` assumes monotonic contrast ordering when scanning for a target step.
- **De-risking combinatorial ugliness** (the real risk of free-form knobs vs. hand-tuned presets): a strong default (the gamut-relative algorithm), named snapshot points along each slider, curated full-equalizer presets (same pattern as the existing Theme Shop, one level deeper), and live preview as the final safety net.
